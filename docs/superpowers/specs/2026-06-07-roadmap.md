### User Stories

| ID | Story | Acceptance Criteria |
|---|---|---|
| I18N-1 | Là người dùng, tôi thấy giao diện bằng tiếng Việt khi mở app lần đầu | Tất cả label, placeholder, status message hiển thị tiếng Việt |

## Phase 2 — Real Ollama Chat (Streaming)

**Status:** Not started  
**Depends on:** Phase 1  
**Estimated complexity:** Medium (4–6 ngày)

### Mục tiêu
Thay thế stub agent bằng kết nối thực tới Ollama. Người dùng chat với LLM local, phản hồi stream từng token theo kiểu terminal.

### User Stories

| ID | Story | Acceptance Criteria |
|---|---|---|
| CHAT-1 | Là developer, tôi chat với LLM local (llama3.2, codestral) qua giao diện terminal | Gửi message → Ollama trả lời thật, không phải stub |
| CHAT-2 | Là developer, tôi thấy AI trả lời stream từng token như gõ terminal | Tokens xuất hiện lần lượt, không đợi cả response |
| CHAT-3 | Là developer, tôi chọn model muốn dùng từ danh sách model có sẵn | Dropdown hiện danh sách từ `GET /api/ollama/models`, model được lưu vào session |
| CHAT-4 | Là người dùng, tôi thấy trạng thái rõ ràng khi Ollama offline | Status indicator đổi màu đỏ, message lỗi thân thiện |
| CHAT-5 | Là người dùng, tôi có thể dừng generation đang chạy | Nút "Dừng" (◼) xuất hiện khi agent đang generate |

### Phạm vi kỹ thuật

**Backend (`AgentModule`):**
- `GET /api/ollama/models` → proxy đến `http://host.docker.internal:11434/api/tags`
- `POST /api/agent/chat` → gọi Ollama `POST /api/chat` với `stream: true`
- Stream response về frontend qua **Server-Sent Events (SSE)** — NestJS `@Sse()` decorator
- Abort controller để hủy generation khi client disconnect

**Frontend (`ChatPanel`):**
- Thay `fetch()` bằng `EventSource` để nhận SSE stream
- `ModelSelector` component nhỏ trong header ChatPanel
- Spinner `⟳ thinking...` trước khi token đầu tiên đến
- Nút `◼ Dừng` (`AbortController`) khi đang stream
- Tự động gọi `GET /api/ollama/models` khi mount, fallback nếu Ollama offline

**Dữ liệu:**
- Không lưu lịch sử chat vào DB ở phase này (session-only)
- Model selection lưu trong `localStorage` key `workspace.model`

### Out of Scope
- Chat history persistence
- Multi-turn context window management (system prompt mặc định, không có memory)
- Function calling / tool use (Phase 6)

## Phase 3 — Task Kanban Board (Real-time)

**Status:** Not started  
**Depends on:** Phase 1  
**Estimated complexity:** Medium (4–5 ngày)

### Mục tiêu
View Kanban đầy đủ cho bảng `tasks` (đã có từ Phase 0). Board tự cập nhật real-time khi AI agent hoặc user thay đổi task.

### User Stories

| ID | Story | Acceptance Criteria |
|---|---|---|
| TASK-1 | Là người dùng, tôi xem task theo 4 cột: Cần làm / Đang làm / Hoàn thành / Thất bại | Board hiển thị đúng 4 cột theo status TODO/PROCESSING/DONE/FAILED |
| TASK-2 | Là người dùng, tôi kéo task sang cột khác để đổi status | Drag-drop cập nhật status ngay, gọi `PATCH /api/tasks/:id` |
| TASK-3 | Là người dùng, tôi tạo task mới bằng form inline trong cột "Cần làm" | Click `+` → input inline → Enter → task xuất hiện |
| TASK-4 | Là developer, board tự cập nhật khi AI agent thay đổi task | Socket.io event `task:updated` → board re-render, không cần reload |
| TASK-5 | Là người dùng, tôi xóa task hoặc đổi priority từ card menu | Right-click hoặc `...` menu trên card |
| TASK-6 | Là người dùng, tôi lọc task theo priority | Toggle filter priority: Cao / Trung bình / Thấp |

### Phạm vi kỹ thuật

**Backend:**
- Thêm `@nestjs/websockets` + `socket.io` vào backend
- `TasksGateway`: emit `task:created`, `task:updated`, `task:deleted` sau mỗi mutation
- `TasksService` gọi gateway sau mỗi CRUD operation

**Frontend:**
- `TasksView.vue` — hiển thị khi click icon "Nhiệm vụ" trong sidebar
- `KanbanBoard.vue` — 4 cột, dùng `vue-draggable-plus` cho drag-drop
- `TaskCard.vue` — card đơn lẻ với title, priority badge, due date
- `socket.io-client` kết nối tới backend, lắng nghe events trên `namespace /tasks`
- AppShell: khi `activeView === 'tasks'`, `TasksView` chiếm toàn bộ content area (cả ChatPanel + ArtifactsPanel đều ẩn) vì Kanban cần full-width

### Out of Scope
- Sub-tasks / checklist trong task
- Task assignment (single user app)
- Due date calendar picker (text input đủ ở phase này)

## Phase 4 — Settings Panel

**Status:** Not started  
**Depends on:** Phase 2  
**Estimated complexity:** Small (2–3 ngày)

### Mục tiêu
Cho phép cấu hình workspace mà không cần sửa file `.env` hay restart container.

### User Stories

| ID | Story | Acceptance Criteria |
|---|---|---|
| SET-1 | Là developer, tôi đổi Ollama base URL nếu chạy Ollama trên máy khác | URL được lưu, `AgentService` dùng URL mới cho request tiếp theo |
| SET-2 | Là developer, tôi chọn model mặc định từ danh sách Ollama | Persist vào `settings` table SQLite, load khi app khởi động |
| SET-3 | Là người dùng, tôi đổi ngôn ngữ từ Settings | Giống nút toggle sidebar nhưng trong form settings |
| SET-4 | Là người dùng, tôi xem thông tin app: version, trạng thái backend, DB | Read-only info panel |

### Phạm vi kỹ thuật

**Backend:**
- `settings` Prisma model: `{ key: String @unique, value: String }`
- `SettingsModule`: `GET /api/settings`, `PATCH /api/settings/:key`
- Built-in keys: `ollama.baseUrl`, `ollama.defaultModel`

**Frontend:**
- `SettingsView.vue` — form settings, hiển thị khi click icon ⚙️
- Đọc current settings từ `GET /api/settings` khi mount
- Save button → `PATCH /api/settings/:key` cho từng key thay đổi
- Hiển thị version app từ `package.json` + health status

### Out of Scope
- User accounts / authentication
- API key management cho external LLM (Phase sau)
- Export/import settings

## Phase 5 — Knowledge Base & RAG

**Status:** Not started  
**Depends on:** Phase 2 + Phase 4  
**Estimated complexity:** Large (8–10 ngày)

### Mục tiêu
Cho phép AI agent truy cập ngữ cảnh từ tài liệu và codebase local. Người dùng upload file hoặc chỉ định thư mục codebase → AI tự động tìm kiếm context liên quan khi trả lời.

### User Stories

| ID | Story | Acceptance Criteria |
|---|---|---|
| RAG-1 | Là developer, tôi upload PDF/DOCX/TXT và hỏi AI về nội dung | File được index, câu trả lời dựa trên nội dung file |
| RAG-2 | Là developer, tôi trỏ AI vào thư mục codebase và hỏi về architecture | Backend watch thư mục, chunk code, embed, lưu vào LanceDB |
| RAG-3 | Là người dùng, tôi xem danh sách tài liệu đã index trong Files view | Files tab hiển thị list với tên file, kích thước, ngày index |
| RAG-4 | Là người dùng, tôi xóa tài liệu khỏi knowledge base | File bị xóa khỏi LanceDB và storage |
| RAG-5 | Là developer, AI tự động dùng RAG khi câu hỏi liên quan đến code/tài liệu | Vector search top-3 chunks được inject vào system prompt |

### Phạm vi kỹ thuật

**Backend:**
- `@lancedb/lancedb` — embedded vector DB trong `/app/data/lancedb/`
- `KnowledgeModule`: upload, chunk, embed, search
- Embedding qua Ollama `POST /api/embeddings` (model `nomic-embed-text`)
- File chunking: `langchain/text_splitter` hoặc custom (chunk size 512 tokens, overlap 50)
- File storage: `/app/data/uploads/` (volume mount đã có)
- Support formats: `.pdf` (pdf-parse), `.docx` (mammoth), `.txt`, `.md`, `.ts`, `.js`, `.py`
- RAG pipeline trong `AgentService`: query → embed → LanceDB search → inject context → Ollama

**Frontend:**
- `FilesView.vue` — hiển thị khi click icon "Tệp tin"
- Upload zone (drag-drop + click) với progress indicator
- Danh sách file đã index với status (indexing / ready / error)
- Codebase watcher: người dùng mount thư mục codebase vào container qua `docker-compose.yml` (`volumes: - /path/to/myproject:/workspace:ro`), sau đó nhập đường dẫn `/workspace` trong UI → backend watch & index

### Out of Scope
- XLSX support (complex parsing, Phase sau)
- Hybrid search (BM25 + vector)
- Re-ranking
- Per-document access control

## Phase 6 — Agent Tools & Agentic Automation

**Status:** Not started  
**Depends on:** Phase 2 + Phase 3 + Phase 5  
**Estimated complexity:** Large (7–10 ngày)

### Mục tiêu
AI agent có thể tự chủ thực thi actions: tạo task, cập nhật task, tìm kiếm knowledge base — trong khi hội thoại với người dùng. Hành động của agent được stream live lên UI.

### User Stories

| ID | Story | Acceptance Criteria |
|---|---|---|
| AGT-1 | Là developer, tôi nói "Tạo task cài đặt auth module priority cao" → agent tự tạo task | Task xuất hiện trong Kanban Board, agent xác nhận đã tạo |
| AGT-2 | Là developer, agent tìm kiếm knowledge base khi cần ngữ cảnh | Agent tự gọi tool `search_knowledge`, inject kết quả vào response |
| AGT-3 | Là developer, tôi thấy execution log của agent trong real-time | ChatPanel hiển thị `[tool] create_task(...)` → `[result] Task #5 created` |
| AGT-4 | Là developer, tôi xem lịch sử tool calls trong ArtifactsPanel | Artifacts panel log tất cả tool calls và kết quả |
| AGT-5 | Là người dùng, agent báo cáo khi không thể thực thi tool | Error message rõ ràng thay vì im lặng fail |

### Phạm vi kỹ thuật

**Backend:**
- Tool schema định nghĩa theo Ollama function calling format
- Tools: `create_task(title, priority, description)`, `update_task(id, status)`, `list_tasks(status?)`, `search_knowledge(query)`
- `AgentService` parse Ollama tool_call response → execute tool → feed result back → continue stream
- Socket.io emit `agent:tool_call` và `agent:tool_result` để frontend hiển thị live

**Frontend:**
- `ChatPanel` render tool call events với icon riêng (`[⚙]` prefix)
- `ArtifactsPanel` tab "Agent Log" hiển thị timeline tool calls
- Phân biệt visual: user message / agent text / tool call / tool result / system message

### Out of Scope
- Multi-agent orchestration
- Agent memory persistence (long-term)
- Custom tool definition by user (UI)
- Scheduled/autonomous agent runs

## Nguyên tắc áp dụng cho mọi Phase

Theo `CLAUDE.md` — bắt buộc tuân thủ:

1. **UI**: Chỉ dùng Tailwind color tokens (`cyber-*`), `font-mono`, max `rounded`, không shadow, không gradient.
2. **Security**: DOMPurify cho mọi `v-html`, Prisma typed queries, typed DTOs, không `origin: '*'`.
3. **i18n**: Mọi chuỗi UI qua `t('key')` — không hardcode. Vietnamese primary, English secondary.
4. **TDD**: Viết spec test trước, implement sau (backend). Frontend verify thủ công.
5. **Commit**: Một commit per logical change, prefix `feat:/fix:/chore:`.
